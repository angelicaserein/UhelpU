## 第一类：目录与文件命名混乱

### 问题 1.1：目录命名风格不统一

前端项目中，文件夹命名**不应使用 PascalCase**（PascalCase 是类名的约定）。主流规范是 **kebab-case（短横线分隔）**，原因是：
- URL 友好，跨操作系统兼容性最好（macOS 区分大小写，Windows 不区分）
- npm 包名、主流框架（Vue、React）的项目模板均采用此风格

当前项目的 `js/` 下，目录命名风格存在三种混用：

| 目录名 | 当前风格 | 问题 | 建议改名 |
|--------|---------|------|---------|
| `CharacterControlSystem/` | PascalCase | ❌ 文件夹不用驼峰 | `character-control-system/` |
| `CollideSystem/` | PascalCase | ❌ 同上 | `collision-system/` |
| `GameEntityModel/` | PascalCase | ❌ 同上 | `game-entity-model/` |
| `PhysicsSystem/` | PascalCase | ❌ 同上 | `physics-system/` |
| `RecordSystem/` | PascalCase | ❌ 同上 | `record-system/` |
| `KeyBindingSystem/` | PascalCase | ❌ 同上 | `key-binding-system/` |
| `MechanismSystem/` | PascalCase | ❌ 同上 | `mechanism-system/` |
| `EventSystem/` | PascalCase | ❌ 同上 | `event-system/` |
| `GameRuntime/` | PascalCase | ❌ 同上 | `game-runtime/` |
| `ui/` | 小写 | ✅ 风格正确，保持 | `ui/` |
| `switchers/` | 小写 | ✅ 风格正确，保持 | `switchers/` |
| `each-level-design/` | kebab-case | ✅ 格式正确，但命名不达意 | `level-design/` |

**注意**：JS 文件名（如 `RecordSystem.js`、`Player.js`）每个文件对应一个类，保持 **PascalCase** 是合理的，不需要改。改的只是**文件夹名**。

```
js/GameEntityModel/Signboard.js    ← 游戏实体（继承 GameEntity）
js/ui/components/Signboard.js      ← UI 组件（另一个同名文件）
```

两个文件同名且功能相近，极易造成混淆。应当：
- 保留 `GameEntityModel/Signboard.js`（游戏实体逻辑）
- 若 `ui/components/Signboard.js` 是纯渲染组件，应改名为 `SignboardDisplay.js` 或合并进实体类

### 问题 4.4：`BaseLevel.js` 手动管理子系统生命周期

**文件**：`js/each-level-design/BaseLevel.js`

```javascript
// BaseLevel.js 第15-29行

    if (this.recordSystem && typeof this.recordSystem.clearAllListenersAndTimers === "function") {
        this.recordSystem.clearAllListenersAndTimers();
    }
    const player = this.referenceOfPlayer();
    if (player && typeof player.clearListeners === "function") {
        player.clearListeners();
```

```javascript
clearLevel() {
    for (const system of this.systems) {
        system.destroy?.();
    }
    this.entities.clear();
}
```

```javascript
    for (const dyn of this.dynamicEntities) {
        for (const sta of this.staticEntities) {
            this.processDynamicStaticPair(dyn, sta);
        }
    }

    // 第2次 Dynamic-Dynamic（同样的代码）
    if (replayer && replayer.isReplaying) {
        this.processDynamicDynamicPair(this.getPlayer(), replayer);
    }

    // Dynamic-Static（再一次）
    for (const dyn of this.dynamicEntities) {
        for (const sta of this.staticEntities) {
            this.processDynamicStaticPair(dyn, sta);
        }
    }

    // 第3次 Dynamic-Dynamic（再一次）
    if (replayer && replayer.isReplaying) {
        this.processDynamicDynamicPair(this.getPlayer(), replayer);
    }
    // ...
}
```

这三次重复虽然是有意为之（多轮碰撞修正），但没有提取成辅助方法，可读性极差。

**建议**：
```javascript
collisionEntry(eventBus) {
    const PASSES = 3;
    for (let i = 0; i < PASSES; i++) {
        this._resolveDynamicDynamic();
        this._resolveDynamicStatic();
    }
    this._processTriggers(eventBus);
}
```

## 第六类：模块耦合与架构缺陷

### 问题 6.1：循环依赖链

**涉及文件**：`GameEntityModel/Signboard.js`、`each-level-design/BaseLevel.js`、`ui/pages/game-pages/GamePageLevel1.js`

```
Signboard.js 持有 Level 引用（通过构造函数注入）
     ↓ 调用 this.level.referenceOfPlayer()
BaseLevel.js 通过 EventBus 发送 signboardInteracted 事件
     ↓
GamePageLevel1.js 订阅该事件并更新 UI
     ↓
GamePageLevel1 又通过 switcher.eventBus 与 Level 通信
```

**证据** — `Signboard.js` 第 23 行：
```javascript
constructor(x, y, w, h, level = null, eventBus = null) {
    this.level = level;   // 直接持有 Level 引用！
}

// 第99行：通过 Level 获取 Player
isPlayerOverlapping() {
    const player = this.level?.referenceOfPlayer();
    ...
}
```

游戏实体不应该持有关卡（Level）的引用，这违反了依赖方向。

**建议**：Signboard 通过 EventBus 请求玩家位置信息，或在每帧 update 时由 Level 主动将玩家位置注入（依赖注入）：
```javascript
// BaseLevel.js 中
update(p) {
    const player = this.referenceOfPlayer();
    for (const entity of this.entities) {
        entity.update?.(p, { playerX: player.x, playerY: player.y });
    }
}
```

### 问题 6.2：全局暂停状态存在两个来源，容易不同步

**涉及文件**：`GameRuntime/GamePauseState.js`、`each-level-design/LevelManager.js`

**证据 1** — `GamePauseState.js`（全局模块级变量）：
```javascript
let paused = false;
export function setGamePaused(value) { paused = !!value; }
export function isGamePaused() { return paused; }
```

**证据 2** — `LevelManager.js` 第 172-174 行（实例属性 + 同步全局）：
```javascript
setPaused(paused) {
    this.paused = !!paused;
    setGamePaused(this.paused);  // 同时修改全局
}
```

**证据 3** — `RecordSystem.js`、`Clip.js` 等 6+ 个文件直接导入全局：
```javascript
import { isGamePaused } from "../GameRuntime/GamePauseState.js";
// ...
if (isGamePaused()) { return; }
```

**风险**：如果有任何代码只修改了 `LevelManager.paused` 而忘记调用 `setGamePaused()`，或者直接修改了 `GamePauseState` 而绕过了 `LevelManager`，状态就会不同步。

**建议**：删除 `LevelManager.paused`，所有暂停读写都通过 `GamePauseState`（或重命名为 `GameStateManager`）统一管理。

### 问题 6.3：`Level1PromptState.js` 是难以测试的全局状态

**文件**：`js/GameRuntime/Level1PromptState.js`

```javascript
// 全局模块级变量（无法重置，无法 mock）
let hasOpenedRecordHudUi = false;
let hasEnteredReplayState = false;

export function markLevel1RecordHudOpened() { hasOpenedRecordHudUi = true; }
export function shouldShowLevel1MissedPrompt() { return !hasOpenedRecordHudUi; }
export function markLevel1ReplayStarted() { hasEnteredReplayState = true; }
export function shouldShowLevel1ReplayPrompt() { return hasEnteredReplayState; }
export function resetLevel1PromptState() {
    hasOpenedRecordHudUi = false;
    hasEnteredReplayState = false;
}
```

- 这类状态应该是 Level1 实例的私有状态，而不是独立的全局模块
- 全局状态在模块加载后持续存在，即使关卡被卸载重新加载，状态也需要手动 reset

**建议**：将这些状态移入 `Level1` 类的实例属性。

### 问题 6.4：`GameEntity` 子类缺少统一生命周期接口

**涉及文件**：`GameEntityModel/` 所有文件

不同实体的方法定义极度不一致：

| 方法 | Player | Replayer | Ground | Wall | Portal | Spike | Signboard |
|------|:------:|:--------:|:------:|:----:|:------:|:-----:|:---------:|
| `update(p)` | ❌ 无 | ❌ 无 | ❌ 无 | ❌ 无 | ❌ 无 | ❌ 无 | ✅ 有 |
| `draw(p)` | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| `createListeners()` | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ | ✅ |
| `clearListeners()` | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ | ✅ |

更严重的问题 —— `Portal.draw()` 在绘制方法中修改了状态：
```javascript
// Portal.js draw() 方法
draw(p) {
    this._frame++;    // ← 在 draw 中修改状态！这违反了绘制方法应是纯副作用的原则
    // ...
}
```

**建议**：在 `GameEntity` 基类中定义明确的生命周期接口：
```javascript
class GameEntity {
    update(p) { }       // 状态更新（包括帧计数等）
    draw(p) { }         // 纯绘制，不修改状态
    onDestroy() { }     // 清理监听器等
}
```
