## 三、游戏核心玩法

### 3.1 Record & Replay 机制

这是游戏的**核心创新机制**：

1. **录制阶段**：玩家站在地面上，按 `E` 键开始录制（最长 5 秒）。所有键盘输入和时间戳被记录到 `Clip` 数据结构中。
2. **生成分身**：录制结束后，系统创建一个 `Replayer`（分身实体），从玩家起点开始，自动重放录制的操作序列。
3. **时空协作**：玩家（当前时刻）与分身（过去的自己）**同时存在于场景中**，可互相踩踏借力，共同解决谜题。
4. **目标**：利用分身和自己的协作，到达关卡终点的 `Portal`（传送门）。

### 3.2 关卡流程

```
主菜单 → 开场故事 → 关卡选择 → 游戏关卡（Level1 / Level2 / Level3）
  ↓（通过传送门）
关卡结果页 → 结局故事 → 制作人员 / 关卡选择
```

## 四、模块职责详解

### 4.1 AppCoordinator（应用协调器）

**文件**：`js/AppCoordinator.js`

整个应用的**总调度中心**，是连接所有子系统的枢纽。

**职责**：
- 初始化并持有 `SwitcherMain`（页面切换）和 `LevelManager`（关卡管理）
- 监听全局事件（`loadLevel`、`unloadLevel`、`autoResult`、`returnLevelChoice` 等）
- 根据事件协调页面切换与关卡加载/卸载
- 管理背景音乐（BGM）随场景的切换播放
- 每帧调用 `updateFrame()`，驱动整个应用更新与绘制

**关键方法**：
```javascript
init()                    // 初始化：绑定事件、显示主菜单
updateFrame()             // 每帧入口（由 p5.draw 调用）
playLevelBgm(levelIndex)  // 播放对应关卡 BGM
```

### 4.2 AssetsManager（资源管理器）

**文件**：`js/AssetsManager.js`

**职责**：
- 在游戏启动时**异步批量加载**所有资源（图片、字体、音频路径数据）

- 支持 p5.js 2.0 的 `async/await` 加载模式
- 提供全局资源访问接口（图片帧、贴图集等）

**职责**：

### 4.4 EventSystem / EventBus（事件总线）

**文件**：`js/EventSystem/EventBus.js`

实现简单的**发布-订阅（Pub/Sub）模式**，用于模块间解耦通信。

**核心 API**：
```javascript
eventBus.publish("loadLevel", "level1")   // 发布事件
eventBus.subscribe("loadLevel", handler)  // 订阅事件
eventBus.unsubscribe("loadLevel", handler) // 取消订阅
```

**主要事件列表**：

| 事件名 | 触发者 | 含义 |
|--------|--------|------|
| `loadLevel` | UI 页面 | 请求加载关卡 |
| `unloadLevel` | 关卡 / UI | 请求卸载关卡 |
| `pauseGame` | 暂停按钮 | 暂停游戏 |
| `resumeGame` | 暂停窗口 | 恢复游戏 |
| `autoResult` | 关卡逻辑 | 自动触发结果（成功/失败） |
| `returnLevelChoice` | 结果页 | 返回关卡选择 |
| `signboardInteraction` | Signboard | 玩家与木牌交互 |

### 4.5 GameEntityModel（游戏实体模型）

**文件夹**：`js/GameEntityModel/`

所有游戏对象的数据模型层，采用**组合优先**的设计模式。

#### 实体继承层次

```
GameEntity（基类）
├── Character
│   ├── Player    —— 玩家主角，监听键盘输入
│   └── Replayer  —— 分身，自动重放录制的输入
├── BasePlatform
│   ├── Ground    —— 可踩踏地面
│   └── Platform  —— 特殊平台
├── Wall          —— 不可穿越的墙体
├── Portal        —— 传送门（触发器，关卡终点）
├── Spike         —— 刺刺（触发器，碰到死亡）
├── Button        —— 按钮（触发器，机制激活）
├── Signboard     —— 木牌（触发器，可交互提示）
├── KeyPrompt     —— 按键提示（纯视觉）
└── TextPrompt    —— 文字提示（纯视觉）
```

#### 实体碰撞体类型

| 实体 | 碰撞类型 | 说明 |
|------|---------|------|
| Player / Replayer | `DYNAMIC` | 受物理影响，参与碰撞解算 |
| Ground / Wall / Platform | `STATIC` | 不动，阻挡 DYNAMIC 实体 |
| Portal / Spike / Button / Signboard | `TRIGGER` | 不阻挡，但触发回调 |
| KeyPrompt / TextPrompt | 无 | 纯 UI 显示 |

### 4.6 CharacterControlSystem（角色控制系统）

**文件夹**：`js/CharacterControlSystem/`

实现了完整的**分层输入处理管线**，将键盘事件转化为物理速度，共 5 个处理层：

```
键盘事件（keydown/keyup）
     ↓
EventProcesser     —— 过滤允许的按键（来自 KeyBindingManager）
     ↓
IntentResolver     —— 将按键映射为玩家意图（wantsJump / wantsLeft / wantsRight 等）
     ↓
ActionValidator    —— 结合能力条件（isOnGround / canJump 等）验证意图，确定有效动作
     ↓
PhysicsApplier     —— 将动作转化为速度/加速度（accX / velY）
     ↓
PhysicsSystem      —— 每帧将加速度积分为速度，速度积分为位移
```

**ControllerManager**（`ControllerManager.js`）：
- 管理单个实体（Player 或 Replayer）的完整控制管线
- 支持多实例（Player 和 Replayer 各自拥有独立的 ControllerManager）
- 每帧调用 `tick()` 处理持续按键状态

**控制模式**（`ControlMode.js`）：
- `BasicMode`：用于 Player，响应真实键盘输入
- `BasicModeReplayer`：用于 Replayer，从 Clip 数据中读取事件自动应用

### 4.7 PhysicsSystem（物理系统）

**文件夹**：`js/PhysicsSystem/`

**文件**：
- `PhysicsSystem.js`：物理引擎主体
- `MovementComponent.js`：运动数据组件

**每帧物理更新流程**：
```
1. 重置能力标志（isOnGround = false）
2. ControllerManager.tick()（处理持续按键，更新加速度）
3. velY += accY（重力加速度累加）
4. velX += accX（水平加速度累加）
5. x += velX；y += velY（位移更新）
```

**MovementComponent 数据结构**：
```javascript
{
  velX,  velY,   // 当前速度（像素/帧）
  accX,  accY    // 当前加速度（含重力）
}
```

### 4.8 CollideSystem（碰撞系统）

**文件夹**：`js/CollideSystem/`

使用 **AABB（轴对齐包围盒）** 碰撞检测，采用**检测-解决-响应**三阶段设计。

#### 碰撞检测轮次（每帧执行多次，防止穿透）

```
第1轮：Dynamic-Dynamic（Replayer 推 Player）
第2轮：Dynamic-Static（角色与地形碰撞）
第3轮：Dynamic-Dynamic（再次防重叠）
第4轮：Dynamic-Static（防被挤入平台）
第5轮：Dynamic-Dynamic（帧末最终检查）
第6轮：Dynamic-Trigger（触发器检测：Portal/Spike/Button/Signboard）
```

**三个映射表**：
- `detectorMap.js`：根据碰撞对类型选择检测算法
- `resolverMap.js`：根据碰撞结果计算位移修正量
- `responderMap.js`：根据碰撞类型触发对应回调（如 Spike → 死亡，Portal → 过关）

### 4.9 RecordSystem（录制系统）

**文件夹**：`js/RecordSystem/`

游戏核心机制的实现层。

**文件职责**：

| 文件 | 职责 |
|------|------|
| `RecordSystem.js` | 录制/重放主体，管理录制状态机 |
| `RecordUI.js` | Steampunk 风格 HUD，显示录制状态和进度条 |
| `RecordPlayState.js` | 重放状态枚举（Standby / Recording / ReadyToReplay / Replaying） |
| `Clip.js` | 录制数据结构（带时间戳的事件序列） |
| `RecordKeyUtil.js` | 按键事件工具函数 |

**录制流程**：
```
玩家站在地面 → 按 E 键 → startRecording()
  → 每帧记录 {frameIndex, eventType, keyCode} 到 Clip
  → 5秒上限或再次按 E → stopRecording()
  → 创建 Replayer 实体，从起点开始重放 Clip
  → Player 和 Replayer 同时进行物理/碰撞更新
```

### 4.10 KeyBindingSystem（按键绑定系统）

**文件夹**：`js/KeyBindingSystem/`

**文件**：
- `KeyBindingManager.js`：单例，提供意图↔按键的双向查询接口
- `KeyBindingConfig.js`：配置读写，持久化到 `localStorage`

**核心接口**：
```javascript
const km = KeyBindingManager.getInstance()
km.getIntentByKey("KeyW")      // → "jump"
km.getKeyByIntent("jump")      // → "KeyW"
km.getAllowedKeys()             // → Set<keyCode>（用于过滤输入）
km.rebind("jump", "Space")     // 重新绑定
km.reset()                     // 恢复默认
km.onChange(callback)          // 监听配置变更
```

### 4.12 MechanismSystem（机制系统）

**文件夹**：`js/MechanismSystem/`

**文件**：`WireIndicatorSystem.js`

Level2 的**电线传导机制**实现：

- 接受一组 `Button` 实体和对应的电线路径点
- 当所有按钮同时被按压时，触发电弧动画沿电线路径传导
- 路径终点有**指示灯**，全亮后永久开启 Portal（传送门）
- 电弧动画：背景灰色路径 + 蓝色发光电弧 + 锯齿抖动效果

**构造方法签名**：
```javascript
new WireIndicatorSystem(
  buttons,          // Button[] 按钮列表
  wirePaths,        // Vec2[][] 每条电线的路径点数组
  indicatorPositions, // Vec2[] 指示灯位置
  onAllArrived,     // () => void 全部到达时的回调
  { wireSpeed }     // 配置（传导速度）
)
```

```javascript
class BaseLevel {
  entities: Set<GameEntity>          // 所有游戏实体
  physicsSystem: PhysicsSystem       // 物理引擎
  collisionSystem: CollisionSystem   // 碰撞系统
  recordSystem: RecordSystem         // 录制系统
}
```

**关键方法**：
```javascript
loadLevel()                    // 创建实体、初始化系统、绑定事件
unloadLevel()                  // 清理所有实体和监听器
updatePhysics()                // 每帧物理更新
updateCollision(p, eventBus)   // 每帧碰撞检测
draw(p)                        // 绘制所有实体
clearCanvas()                  // 绘制背景
addReplayer(startX, startY)    // 动态添加分身
removeReplayer()               // 移除分身
```

#### Level1（教学关卡）

**主题**：教学，让玩家学习基本移动和 Record & Replay 操作。

**地形设计**：
- 玩家起点在左下方（x=50, y=450）
- 右侧有一个高平台（200×120），放置传送门
- 场景中有 Signboard 木牌，引导玩家理解机制
- KeyPrompt 显示 WAD 按键提示

**特殊逻辑**：
- 玩家靠近木牌时高亮显示
- 按 `E` 键与木牌交互，弹出故事文本窗口
- `Level1PromptState` 追踪玩家进度，动态调整提示内容

#### Level2（电力机制关卡）

**主题**：双角色协作，触发电力机制开启传送门。

**谜题设计**：
- 场景中有**两个按钮**（左/右各一个）
- 两个按钮必须**同时被按住**（玩家和分身各站一个）
- 电力沿电线路径传导到指示灯
- 指示灯全亮 → 传送门永久开启

**WireIndicatorSystem** 在此关卡被实例化和使用。

#### Level3

高级关卡，机制更复杂（待开发完善）。

#### LevelManager（关卡管理器）

**职责**：
- 根据关卡索引实例化对应的 Level 类（Level1 / Level2 / Level3）
- 调用 `loadLevel()` / `unloadLevel()`
- 每帧驱动关卡的 `update()` 和 `draw()`

#### Room（房间/屏幕）

**职责**：实现**多屏系统**（关卡可由多个 Room 组成，每个 Room 是一个独立的视口）。支持玩家在房间之间切换（类似老式横版游戏的屏幕切换）。

## 五、完整数据流示例

### 从启动到进入 Level1 的完整流程

```
1. 浏览器加载 index.html
   └→ 引入 p5.js 和 main.js

2. main.js 执行 p5 实例化
   └→ p5.setup():
       ├→ 创建 1280×720 画布
       ├→ await Assets.loadAll(p)  （异步加载所有资源）
       └→ app = new AppCoordinator(p)
          └→ app.init():
              ├→ 创建 SwitcherMain
              ├→ 创建 LevelManager
              ├→ 绑定全局事件（loadLevel / unloadLevel / autoResult ...）
              └→ SwitcherStaticPage.showMainMenu()  显示主菜单

3. 玩家点击 "PLAY" 按钮
   └→ StaticPageMenu → 跳转 StaticPageOpeningStory
      └→ AudioManager.playBGM("openingStory")

4. 开场动画结束 / 玩家跳过
   └→ 跳转 StaticPageLevelChoice  （关卡选择页）

5. 玩家点击 Level1 按钮
   └→ eventBus.publish("loadLevel", "level1")
      └→ AppCoordinator 监听处理:
          ├→ AudioManager.playBGM("level1")
          ├→ LevelManager.loadLevel("level1", p, eventBus)
          │   └→ new Level1()
          │       ├→ 创建所有实体（Player, Ground, Wall, Signboard, Portal...）
          │       ├→ 初始化 PhysicsSystem / CollisionSystem / RecordSystem
          │       └→ Player.createListeners()  监听键盘事件
          └→ 创建 GamePageLevel1 UI，切换到游戏视图

6. p5.draw() 每帧循环（60fps）
   └→ app.updateFrame():
       ├→ SwitcherMain.update() + draw()  （UI 层）
       └→ LevelManager.update(p, eventBus):
           ├→ Level.updatePhysics():
           │   ├→ PhysicsSystem.physicsEntry()
           │   │   ├→ ControllerManager.tick()  处理持续按键
           │   │   └→ 积分加速度→速度→位置
           │   └→ 更新 Signboard 交互检测
           │
           ├→ Level.updateCollision():
           │   ├→ Dynamic-Dynamic 碰撞（Player ↔ Replayer）
           │   ├→ Dynamic-Static 碰撞（角色 ↔ 地形）
           │   └→ Dynamic-Trigger 碰撞（Portal/Spike/Button/Signboard）
           │
           └→ Level.draw():
               ├→ clearCanvas()  绘制背景
               ├→ 绘制所有 Static 实体（地形）
               ├→ 绘制所有 Dynamic 实体（Player/Replayer）
               └→ 绘制所有 UI 实体（KeyPrompt/TextPrompt）
```

### 录制与重放数据流

```
玩家在地面按 E 键
  └→ RecordSystem.startRecording()
      └→ state = RECORDING
         每帧: clip.record({frame, eventType, keyCode})

再次按 E 键 / 5秒超时
  └→ RecordSystem.stopRecording()
      └→ state = READY_TO_REPLAY
         level.addReplayer(startX, startY)
           └→ new Replayer(startX, startY)
               └→ 绑定独立的 ControllerManager（BasicModeReplayer）

每帧重放:
  └→ ControllerManager(Replayer).tick()
      └→ 从 Clip 读取当前帧的事件
         └→ 应用到 Replayer 的 movementComponent
            └→ PhysicsSystem 同等处理 Replayer 的物理
               └→ CollisionSystem 同等处理 Replayer 的碰撞
```

## 七、关键设计模式

| 模式 | 应用位置 | 说明 |
|------|---------|------|
| 发布-订阅（Pub/Sub） | EventBus | 模块间解耦通信 |
| 单例（Singleton） | KeyBindingManager, AssetsManager | 全局唯一实例 |
| 组合（Composition） | GameEntity + 各 Component | 避免深层继承，灵活组合能力 |
| 状态机（State Machine） | RecordPlayState, GamePauseState | 管理游戏状态转换 |
| 管线（Pipeline） | CharacterControlSystem | 输入→意图→动作→物理 分层处理 |
| 策略（Strategy） | detectorMap/resolverMap/responderMap | 根据碰撞对类型选择不同算法 |
| 页面对象（Page Object） | PageBase + 各 StaticPage/GamePage | 统一页面生命周期接口 |

## 八、总结

**UhelpU** 是一个设计精良的 2D 益智平台游戏原型，架构清晰，具备以下特点：

1. **清晰的职责分离**：物理、碰撞、录制、UI、音频各系统独立，互不耦合
2. **完整的控制管线**：输入→意图→动作→物理的分层处理，易于扩展新控制模式
3. **核心机制完整实现**：Record & Replay 机制完全可用，分身与玩家同步物理运算
4. **灵活的 UI 架构**：静态页面和游戏页面分离，生命周期统一管理
5. **国际化支持**：中英文双语，语言切换即时生效
6. **可配置性**：按键绑定、音量等均持久化，支持玩家自定义
7. **可扩展的关卡系统**：基于 BaseLevel 可快速实现新关卡和新机制
