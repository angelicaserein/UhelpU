## 1. 系统概览

排行榜系统支持两种玩家身份：**游客**和**注册账号**，两者都可以上传成绩、显示在排行榜上。

核心特性：

- 每个关卡独立排行榜，成绩不混用
- 每个玩家只显示最佳成绩（去重）
- 同时显示本次成绩、本次排名、历史最佳
- 注册用户排行榜名字前显示 👑 标识
- 游客和账号用户命名空间完全独立

## 2. 技术选型

### 为什么用 Firebase

项目是纯前端 p5.js 游戏，部署在 GitHub Pages，没有后端服务器。Firebase 提供免费的云数据库，前端可以直接调用，不需要搭建服务器。

### 为什么用 REST API 而不是 Firebase SDK

Firebase 官方 SDK 使用 ES Module 的 `import` 语法，与项目现有的 p5.js 加载方式存在冲突（`Unexpected token 'export'` 报错）。因此改用 Firestore REST API，直接用 `fetch` 调用，没有任何依赖。

### Firebase 免费额度（Spark Plan）

| 资源 | 免费额度 |
|------|----------|
| 每日读取 | 50,000 次 |
| 每日写入 | 20,000 次 |
| 存储空间 | 1 GB |

对学生项目完全够用。

## 3. 数据库结构

### Firestore 集合总览

```
Firebase Project: uhelpu
│
├── leaderboard/                    # 排行榜数据
│   └── {levelId}/                  # 关卡ID，如 easy_level2
│       └── scores/                 # 该关卡所有成绩
│           └── {docId}             # 每条成绩记录
│
├── playerNames/                    # 游客名字去重计数
│   └── {baseName}                  # 文档ID为名字小写，如 moosry
│
├── accountNames/                   # 账号用户名去重计数
│   └── {baseName}                  # 文档ID为用户名小写
│
└── users/                          # 注册用户信息
    └── {uid}                       # 文档ID为 Firebase Auth uid
```

### leaderboard 成绩记录字段

```json
{
  "playerName": "moosry",
  "timeMs": 7120,
  "timeSeconds": "7.12",
  "levelId": "easy_level2",
  "isAccount": false,
  "timestamp": "2026-04-01T12:00:00.000Z",
  "submittedAt": "2026-04-01T12:00:00.000Z"
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| playerName | string | 玩家显示名字（含编号，如 moosry#2） |
| timeMs | integer | 通关时间，毫秒，越小越好 |
| timeSeconds | string | 通关时间，秒，保留两位小数 |
| levelId | string | 关卡标识，如 easy_level2 |
| isAccount | boolean | 是否为注册账号用户 |
| timestamp | timestamp | 提交时间 |

### playerNames / accountNames 字段

```json
{
  "count": 3
}
```

文档ID为名字的小写形式，`count` 表示有多少玩家用过这个名字。

### users 字段

```json
{
  "username": "moosry",
  "email": "player@example.com"
}
```

## 4. 玩家身份系统

### localStorage 存储结构

| 玩家类型 | localStorage 键值 |
|----------|-------------------|
| 游客 | `playerName: "moosry"` |
| 注册账号 | `playerName: "moosry"` + `playerAccount: "{uid, username, email, isAccount: true}"` |

`playerName` 始终是当前显示名字，游客和账号都有。`playerAccount` 只有注册账号用户才有，用于判断当前是否以账号身份登录。

### 启动时自动跳过逻辑

```
游戏启动
    ↓
检查 localStorage 有没有 playerAccount
    ↓ 有
设置 window.playerName = username，跳转 WorldSelect（账号模式）
    ↓ 没有
检查 localStorage 有没有 playerName
    ↓ 有
设置 window.playerName，跳转 WorldSelect（游客模式）
    ↓ 没有
显示 NameInputPage，玩家选择身份
```

### 名字唯一性机制

由于没有密码验证，游客名字通过以下方式处理重名：

1. 玩家输入名字后，查询 `playerNames` 集合中该名字的 `count`
2. 如果 count = 0，直接使用，count 设为 1
3. 如果 count > 0，弹出对话框「前方已有 X 个 [名字]，你确定还要用这个名字吗？」
4. 玩家选择「就要用！」后，count 加 1，名字后面加编号（如 `moosry#2`）
5. localStorage 和 Firebase 均存储带编号的完整名字

## 5. 排行榜核心逻辑

### 成绩提交流程

```
玩家通关
    ↓
记录通关时间（毫秒）
    ↓
读取 localStorage 的 playerName 和 isAccount
    ↓
POST 到 Firestore leaderboard/{levelId}/scores
    ↓
提交完成，加载排行榜数据
```

### 排行榜读取流程

由于 Firestore REST API 的 runQuery 排序查询在子集合路径下存在兼容性问题，系统使用降级方案：

```
GET /leaderboard/{levelId}/scores?pageSize=100
    ↓
获取该关卡所有成绩（最多100条）
    ↓
本地按 timeMs 升序排序
    ↓
去重：同名字 + 同 isAccount 类型，只保留最短时间那条
（游客 moosry 和账号 moosry 是两个不同的人，分别保留）
    ↓
取前10名返回给页面显示
```

### 去重逻辑

去重的 key 是 `playerName + "_" + isAccount`，而不是只用 `playerName`。

这样游客 `moosry`（isAccount: false）和账号 `moosry`（isAccount: true）会被视为两个不同的人，各自保留最佳成绩，互不影响。

### WinPage 显示内容

通关结算页显示三行统计信息：

| 信息 | 说明 |
|------|------|
| 本次成绩 | 这次通关的时间，格式 X.XXs |
| 本次排名 | 本次成绩和所有人最佳成绩（去重后）对比的排名 |
| 历史最佳 | 当前玩家该关卡的历史最短时间及对应排名 |

### 排行榜视觉效果

- 第1名：🥇 金色高亮
- 第2名：🥈 银色高亮
- 第3名：🥉 铜色高亮
- 当前玩家：名字高亮 + `← YOU` 标识
- 注册账号：名字前显示 👑
- 加载中：骨架屏占位，背景框和标题立即显示，数据到了再逐条淡入

## 6. 账号系统

### Firebase Authentication

使用 Firebase Auth REST API，不引入任何 SDK。

**注册接口**
```
POST https://identitytoolkit.googleapis.com/v1/accounts:signUp?key={API_KEY}
Body: { email, password, returnSecureToken: true }
Response: { idToken, localId, email }
```

**登录接口**
```
POST https://identitytoolkit.googleapis.com/v1/accounts:signInWithPassword?key={API_KEY}
Body: { email, password, returnSecureToken: true }
Response: { idToken, localId, email }
```

### 注册流程

```
玩家填写用户名 + 邮箱 + 密码
    ↓
检查 accountNames 集合用户名是否重复
    ↓ 重复
弹出重名确认对话框，分配编号（如 moosry#2）
    ↓ 不重复或确认使用
调用 Firebase Auth 注册接口
    ↓
在 Firestore users/{uid} 创建用户文档
    ↓
accountNames 集合该用户名 count + 1
    ↓
localStorage 写入 playerAccount 和 playerName
    ↓
检查是否有游客记录，有则弹出成绩转移对话框
    ↓
进入 WorldSelect
```

### 登录流程

```
玩家填写邮箱 + 密码
    ↓
调用 Firebase Auth 登录接口，获取 uid
    ↓
从 Firestore users/{uid} 读取 username
    ↓
localStorage 写入 playerAccount 和 playerName
    ↓
进入 WorldSelect
```

### 游客升级账号（成绩转移）

注册成功后，如果 localStorage 里有游客 `playerName`，弹出对话框询问是否转移成绩。

确认后执行：

1. 遍历所有关卡的 leaderboard，找到 `playerName == 游客名字` 且 `isAccount == false` 的记录
2. 把这些记录的 `playerName` 更新为新账号用户名，`isAccount` 更新为 true
3. `playerNames` 集合里该游客名字的 count 减 1（count 为 0 时删除文档）
4. 清除 localStorage 里的 `playerName`（游客记录不再需要）

### 改名规则

| 玩家类型 | 改名查重集合 | 影响的排行榜记录 |
|----------|--------------|------------------|
| 游客 | playerNames | isAccount == false 的记录 |
| 账号用户 | accountNames | isAccount == true 的记录 |

改名后：
- localStorage 里的名字更新
- `window.playerName` 更新
- 对应集合的 count 更新（新名字 +1，旧名字 -1）
- Firebase leaderboard 里对应的记录全部更新（只更新对应类型）

### 登出

清除 localStorage 里的 `playerAccount` 和 `playerName`，`window.playerName` 设为 null，跳转回 NameInputPage。

## 7. 命名空间隔离

游客和账号用户的名字系统完全独立：

```
playerNames 集合          accountNames 集合
（游客专用）               （账号用户专用）
    │                           │
    ├── moosry: count=3         ├── moosry: count=1
    ├── test: count=1           └── alice: count=2
    └── alice: count=2
```

两个集合互不影响，游客叫 `moosry` 和账号用户叫 `moosry` 可以同时存在，在排行榜上是两个不同的人。

排行榜去重 key：`playerName + isAccount`，不会互相覆盖。

## 9. API 接口说明

### window.submitScore(playerName, timeMs, levelId)

提交一条成绩到 Firebase。

| 参数 | 类型 | 说明 |
|------|------|------|
| playerName | string | 玩家名字 |
| timeMs | number | 通关时间，毫秒 |
| levelId | string | 关卡ID，如 easy_level2 |

自动从 localStorage 读取 `isAccount` 字段一并写入。

### window.getLeaderboard(levelId, limitCount)

读取指定关卡的排行榜数据。

| 参数 | 类型 | 说明 |
|------|------|------|
| levelId | string | 关卡ID |
| limitCount | number | 返回条数，默认 10 |

返回数组，每条包含：

```javascript
{
  rank: 1,
  playerName: "moosry",
  timeSeconds: "7.12",
  timeMs: 7120,
  isAccount: false
}
```

## 10. 已知限制

| 限制 | 说明 |
|------|------|
| 游客无法防冒充 | 游客没有密码，任何人可以输入相同名字，只能通过编号区分 |
| 改名历史成绩逐条更新 | 玩家记录越多改名越慢，当前用遍历方式更新 |
| runQuery 排序兼容性 | Firestore REST API 的子集合排序查询存在路径兼容问题，使用本地排序降级方案 |
| 单关卡最多读取100条 | 降级方案每次最多读100条原始记录，超出部分不参与排名计算 |
| 无防刷榜机制 | 测试模式下没有限制提交频率，正式上线需要补充安全规则 |
